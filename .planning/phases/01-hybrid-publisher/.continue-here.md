## Session 2026-03-21: OAuth tenant_id Fix + Router Prefix

### Root Cause #1: 401 Unauthorized
**Problem**: POST /api/v1/publisher/{id}/publish devolvía 401

**Investigation**:
- Usuario OAuth tenía `tenant_id = NULL` en DB
- DTO `PublishVehicleRequest` requiere `tenant_id: UUID` obligatorio
- Frontend mock enviaba `tenant_id: "tenant-001"` (string, no UUID)
- `User.create_oauth()` en línea 148 → `tenant_id = None`

**Fix Applied**:
1. `User.create_oauth()` → `tenant_id = user_id` (cada usuario OAuth es su propio tenant)
2. DB migration → `UPDATE users SET tenant_id = id WHERE tenant_id IS NULL;` (10 usuarios)
3. Alembic migration → `20260317_2323-bf038603639b_backfill_tenant_id_for_oauth_users.py`
4. Frontend mock → `tenant_id: "e1871fb7-cf0e-4374-a4ff-89809adffc4e"`

### Root Cause #2: OAuth Login {"detail":"Not Found"}
**Problem**: Al intentar loguearse con Google, frontend obtenía `{"detail":"Not Found"}`

**Investigation**:
- `auth_router` montado en `/api/auth`
- Frontend esperaba `/api/v1/auth`

**Fix Applied**:
- `main.py` → router prefix cambiado de `/api/auth` → `/api/v1/auth`

<context>

**The OAuth tenant_id journey**:

Multi-tenancia es requerimiento arquitectónico. Todos los datos llevan `tenant_id`.

El problema: OAuth users se creaban con `tenant_id = None`.

Solución: Cada usuario OAuth es su propio tenant al inicio. Posteriormente pueden
ser invitados a otras organizaciones (tenant_id se actualiza al aceptar invitación).

Esto es consistente con el diseño: un usuario puede pertenecer a múltiples organizaciones
a través de memberships, pero tiene un tenant_id "default" para datos propios.

**Router prefix issue**:

Frontend usa `/api/v1/*` para todos los endpoints. El auth_router estaba montado
en `/api/auth` causando 404 en OAuth authorize/callback.

Fix: Cambiar prefix a `/api/v1/auth` para consistencia.

</context>
