## Session 2026-03-22: OAuth Router Architecture + Database Migration

### 1. OAuth Router Architecture Decision
**Problem**: OAuth callbacks were using `/api/v1/auth/oauth/google/callback` but Google Cloud Console had `/api/auth/oauth/google/callback` registered.

**Decision Made**:
- OAuth infrastructure (callbacks) → `/api/auth` (SIN versión) - estable, no cambia
- Auth API endpoints → `/api/v1/auth` (CON versión) - evoluciona con v2, v3...

**Rationale**: OAuth callbacks son infraestructura técnica, no contratos de API. Google Cloud Console solo se configura una vez.

### 2. Frontend Changes Applied
- **`apps/web/src/lib/api/authApi.ts`**: All endpoints using `/api/v1/auth/` (login, register, logout, me, etc.)
- **`apps/web/src/components/auth/OAuthButtons.tsx`**: OAuth callbacks using `/api/auth/oauth/${variant}/authorize`

### 3. Backend Configuration
- **`main.py`**: Already has auth_router mounted on BOTH prefixes (lines 149-158):
  - `/api/auth` → OAuth callbacks
  - `/api/v1/auth` → Auth API endpoints
- **`config.py`**: `google_oauth_redirect_uri` needs to stay as `/api/auth/...`

### 4. Database Issue Discovered
**Error**: `relation "oauth_accounts" does not exist`

**Root Cause**: Database `prosell_dev` is EMPTY. No migrations applied.

**Migration Order Problem**: Existing migrations have incorrect order:
1. Initial schema creates `users.id` as VARCHAR
2. Organizations creates `verified_by` as UUID referencing `users.id` (type mismatch!)
3. UUID conversion migration comes AFTER organizations (wrong order!)

### 5. Database Reset
- Dropped and recreated `prosell_dev` database
- Ready for fresh migration

**The OAuth Architecture Journey**:

Empezamos con un problema simple: "autenticación no funciona, redirige al login".

Investigación reveló:
1. Frontend (`authApi.ts`) estaba usando `/api/auth/` pero backend montaba en `/api/v1/auth/`
2. Cambiamos todo a `/api/v1/auth/` - pero eso rompe Google Cloud Console
3. Usuario: "si lo actualizas voy a tener que actualizarlo en google cloud"
4. Decisión: OAuth callbacks son infraestructura, no API - no se versionan
5. Arquitectura final:
   - `/api/auth/oauth/*` → callbacks (Google Cloud, estable)
   - `/api/v1/auth/*` → API endpoints (versionados, evolucionan)

**The Database Problem**:

Cuando intentamos aplicar migraciones, falló con:
```
foreign key "organizations_verified_by_fkey" cannot be implemented
Key columns "verified_by" and "id" are of incompatible types: uuid and character varying
```

Esto pasa porque:
- Migration 1: `users.id` = VARCHAR
- Migration 2: `organizations.verified_by` = UUID → users.id (FAIL!)
- Migration 3: Convert users.id to UUID (too late!)

**Next Steps**:

1. Crear migración fresh con todo en UUID desde el inicio
2. Aplicar con `alembic upgrade head`
3. Reiniciar API
4. Testear OAuth flow
5. Completar UAT
