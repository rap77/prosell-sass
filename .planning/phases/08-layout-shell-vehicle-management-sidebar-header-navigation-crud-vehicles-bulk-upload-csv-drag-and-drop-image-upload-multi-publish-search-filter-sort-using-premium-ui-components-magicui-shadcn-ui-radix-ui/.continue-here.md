**Brain #3 (UI Design):**
- ✅ Paleta 60-30-10 (blue primary para B2B trust)
- ✅ Sans-serif 16px mínimo, Spacing 4/8px scale
- ✅ 5 estados por componente (default/hover/active/disabled/error)

- ✅ Dark Mode: #121212, 15.8:1 contraste, white semi-transparent overlays
- ✅ Mobile: DataGrid → Cards below 768px, 44x44px touch targets

**Brain #4 (Frontend Architecture):**
- ✅ Layout Shell: Server Component + Compound Components Pattern
- ✅ State Strategy: TanStack Query (server) + Zustand (global/persist) + URL (filters) + useState (ephemeral)
- ✅ DataGrid 1000+ rows: TanStack Virtual MANDATORY, stabilize columns, Web Workers if filtering >50ms
- ✅ Route Groups: (admin)/(seller)/(dealer)/(manager) structure
- ✅ Middleware: Centralized guards for Auth+Role+Tenant
- ✅ Image Upload: Zustand for progress (0-100%), TanStack Query onMutate for optimistic previews
- ⚠️ Anti-patterns: SC-01 (localStorage tokens), JS-01 (race conditions), CSS-01 (animating reflow), RX-04 (props drilling)

**Brain #6 (QA/DevOps):**
- ✅ Backend Coverage: 70/20/10 split (Unit/Integration/E2E), 80% minimum for Vehicle CRUD
- ✅ Bulk Upload Testing: 3 fixtures (Success/Partial/Total Failure), pytest.mark.parametrize for chunks
- ✅ Frontend Testing: DataGrid visibility tests, Layout guards RBAC
- ✅ E2E Strategy: Inversion Thinking (test that Seller CANNOT access /settings/tenant)
- ✅ Performance: SLO <200ms for 1000 rows (p95), focus on P99 not average

**Brain #1 (Product Strategy):**
- ✅ Alineación con core value: Bulk Upload + Image Upload atacan "tiempo de salida al mercado"
- ⚠️ Build Trap Risk: Enfocarse en Outcome, no estética (MagicUI/Cmd+K)
- ❌ Gap Crítico: Vehicle Status visibility (En línea/Pendiente/Vendido) — AGREGADO
- ❌ Gap Crítico: Mobile Camera direct upload — AGREGADO
- 💡 Nice-to-have: Analytics de publicación en DataGrid
- 💡 Cmd+K podría ser over-engineering para inventario <20 autos
- 📊 Prioridad ajustada: MVP (Single Vehicle) → UAT (Bulk + Cmd+K) → Premium (Roles avanzados)
- ⚠️ Riesgo técnico: Next.js 16 + React 19 = versiones futuras (alta factibilidad risk)
- 🎯 OKR Sugerido: "Reducir tiempo de carga de inventario de 15 min a 3 min"

**Brain #7 (Growth/Data):**
- ✅ Métricas SLO <200ms, 80% coverage son necesarias pero no suficientes (Guardrails only)
- ❌ Gap: North Star Metric faltante — "Vehículos publicados/vendidos por semana" — AGREGADO
- ❌ Gap: Tracking Aha Moment (primer inventario visible y correcto) — AGREGADO
- ❌ Gap: Search-to-Action Ratio (¿la búsqueda ayuda o es ruido?) — AGREGADO
- 💡 Power User Curve: Identificar vendedores de "alta velocidad"
- ⚠️ Experimentación B2B: Bajo tráfico = A/B tests difíciles, usar Feature Flags
- ⚠️ Lollapalooza Effect: Combo (upload lento + search falla + grid degrada) = rechazo sistémico

**Brain #5 (Backend Architecture):**
- ✅ FastAPI + SQLAlchemy async facilitan Clean Architecture
- ⚠️ Domain entities NO deben tener decoradores SQLAlchemy — VALIDATE
- ⚠️ Repositories deben depender de interfaces, no DB directa — VALIDATE
- ❌ Gap CRÍTICO: Paginación (Cursor-based) en endpoints de búsqueda — AGREGADO
- ❌ Gap: Error Standardization (code + message consistentes) — AGREGADO
- ❌ Gap: API Versioning (/v1/ en URLs) desde el inicio — AGREGADO
- ⚠️ N+1 Problem: Bulk chunks pueden causar N+1 queries — mitigación con eager loading
- ❌ Gap CRÍTICO: Idempotency Keys para bulk retry safety — AGREGADO
- ⚠️ Backpressure: Redis puede saturarse si uploads > procesamiento
- ⚠️ Broken Access Control (IDOR): Cada query DEBE incluir WHERE tenant_id = :current_tenant_id
- ⚠️ SQL Injection: Usar solo Prepared Statements
