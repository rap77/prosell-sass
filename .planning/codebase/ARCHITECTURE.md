# Architecture

**Analysis Date:** 2026-03-14

## Pattern Overview

**Overall:** Clean Architecture with Dependency Injection and Domain-Driven Design

**Key Characteristics:**
- Strict layering: Domain → Application → Infrastructure
- Domain layer has ZERO external dependencies (pure Python)
- All repositories and ports defined in domain layer as interfaces
- Infrastructure layer implements interfaces (Adapter pattern)
- Event-driven architecture for async operations
- Multi-tenant support on all aggregates (tenant_id)

## Layers

**Domain Layer:**
- Purpose: Pure business logic with zero external dependencies
- Location: `apps/api/src/prosell/domain/`
- Contains: Entities, Value Objects, Domain Events, Port Interfaces, Repository Interfaces, Domain Exceptions
- Depends on: Nothing (only Python standard library + Pydantic)
- Used by: Application layer

**Application Layer:**
- Purpose: Orchestration of use cases and business workflows
- Location: `apps/api/src/prosell/application/`
- Contains: Use Cases, DTOs (Pydantic schemas), Application Ports (service interfaces)
- Depends on: Domain layer only
- Used by: Infrastructure layer (FastAPI routers)

**Infrastructure Layer:**
- Purpose: External integrations and technical implementation details
- Location: `apps/api/src/prosell/infrastructure/`
- Contains: FastAPI routers, SQLAlchemy models & repositories, external services, middleware, database config
- Depends on: Domain and Application layers
- Used by: FastAPI application

## Data Flow

**HTTP Request Flow:**

1. Request arrives at `apps/api/src/prosell/infrastructure/api/main.py` (FastAPI app)
2. Middleware stack processes: Security headers → Rate limiting → CORS
3. Router handler in `apps/api/src/prosell/infrastructure/api/routers/` receives request
4. Router injects dependencies (repositories, services) via `apps/api/src/prosell/infrastructure/api/dependencies.py`
5. Use case is instantiated with injected dependencies
6. Use case executes domain logic with domain entities and repository interfaces
7. Infrastructure layer implements repository interfaces with SQLAlchemy models
8. Database session managed by `apps/api/src/prosell/infrastructure/database/session.py`
9. Response DTO serialized by Pydantic and returned

**Database Session Flow:**

1. FastAPI dependency `get_async_session()` from `apps/api/src/prosell/infrastructure/database/session.py`
2. Session created by `async_sessionmaker` with auto-commit on success
3. Auto-rollback on exception
4. Each repository receives session via constructor injection

**Authentication Flow:**

1. User logs in via `POST /api/auth/login` (router in `apps/api/src/prosell/infrastructure/api/routers/auth_router.py`)
2. `LoginUserUseCase` validates credentials using `IPasswordService` (port interface)
3. `PasswordService` implementation verifies password hash
4. JWT token generated by `IJWTService` (port interface)
5. Token stored as HTTP-only cookie via `apps/api/src/prosell/infrastructure/api/middleware/auth_middleware.py`
6. Subsequent requests include JWT in Authorization header or cookies
7. Middleware validates JWT and injects current user into request

**OAuth Flow:**

1. User initiates OAuth: `GET /api/v1/facebook/authorize`
2. `AuthorizeFacebookAccountUseCase` generates state token
3. State token stored in Redis with 10min TTL
4. User redirected to Facebook login
5. Facebook redirects back to `GET /api/v1/facebook/callback?code=...&state=...`
6. `OAuthCallbackUseCase` validates state token in Redis
7. Exchanges code for access token via `IFacebookMarketplaceOAuthService`
8. Access token encrypted with `IEncryptionService` (AES-256) before storage
9. FacebookAccount entity created/updated in database
10. Scheduled task refreshes tokens 48h before expiry

**State Management:**

- **In-memory:** Application state (use case instances, config)
- **Redis:** State tokens (OAuth), rate limiting counters, cache (via `apps/api/src/prosell/infrastructure/services/redis_service.py`)
- **Database:** Persistent entities (Users, Products, etc.) via SQLAlchemy
- **Domain Events:** UserRegisteredEvent, UserLoggedInEvent, etc. (published but not persisted yet)

## Key Abstractions

**Domain Entities:**
- Purpose: Represent core business objects with mutable state
- Examples: `apps/api/src/prosell/domain/entities/user.py`, `apps/api/src/prosell/domain/entities/product.py`, `apps/api/src/prosell/domain/entities/organization.py`
- Pattern: Extend `DomainModel` (mutable Pydantic model), include factory methods and business logic

**Value Objects:**
- Purpose: Represent immutable domain concepts
- Location: `apps/api/src/prosell/domain/value_objects/`
- Pattern: Extend `ValueObject` (immutable Pydantic model), validated at construction

**Port Interfaces (Dependency Inversion):**
- Purpose: Define contracts for external dependencies without implementation
- Location: `apps/api/src/prosell/domain/ports/` and `apps/api/src/prosell/application/ports/`
- Examples: `IJWTService`, `IPasswordService`, `IEncryptionService`, `IFacebookMarketplaceOAuthService`
- Pattern: Abstract base classes or protocols, implemented in infrastructure layer

**Repository Interfaces:**
- Purpose: Abstract database operations
- Location: `apps/api/src/prosell/domain/repositories/`
- Examples: `IUserRepository`, `IProductRepository`, `IFacebookAccountRepository`
- Pattern: Abstract methods with async signatures, implemented by SQLAlchemy repositories in `apps/api/src/prosell/infrastructure/repositories/`

**Use Cases:**
- Purpose: Implement single business action with clear inputs/outputs
- Location: `apps/api/src/prosell/application/use_cases/`
- Pattern: One class per use case, `execute()` method receives DTO, returns DTO, raises domain exceptions
- Examples: `RegisterUserUseCase`, `LoginUserUseCase`, `OAuthCallbackUseCase`, `CreateProductUseCase`

**DTOs (Data Transfer Objects):**
- Purpose: Define input/output contracts for use cases and API endpoints
- Location: `apps/api/src/prosell/application/dto/`
- Pattern: Pydantic models with validation, separate Request/Response classes

**Domain Events:**
- Purpose: Record significant business occurrences
- Location: `apps/api/src/prosell/domain/events/`
- Pattern: Extend `DomainEvent` (immutable), include timestamp, fired but not yet persisted to event store
- Examples: `UserRegisteredEvent`, `UserLoggedInEvent`, `UserEmailVerifiedEvent`

## Entry Points

**FastAPI Application:**
- Location: `apps/api/src/prosell/infrastructure/api/main.py`
- Triggers: Server startup via `fastapi dev` or `uvicorn`
- Responsibilities: Create FastAPI instance, register routers, setup middleware, setup exception handlers

**FastAPI Routers:**
- Location: `apps/api/src/prosell/infrastructure/api/routers/` (e.g., `auth_router.py`, `product_router.py`)
- Triggers: HTTP requests to specific paths
- Responsibilities: Receive HTTP requests, parse input, call use cases, return responses

**Authentication Middleware:**
- Location: `apps/api/src/prosell/infrastructure/api/middleware/auth_middleware.py`
- Triggers: Every HTTP request
- Responsibilities: Extract JWT from cookies/headers, validate signature, inject current user into request

**Database Session Dependency:**
- Location: `apps/api/src/prosell/infrastructure/database/session.py`
- Triggers: Every use case that needs database access
- Responsibilities: Create async session, handle commit/rollback, cleanup

**Scheduled Tasks (Worker):**
- Location: `apps/api/src/prosell/infrastructure/tasks/worker.py`
- Triggers: Background task broker (Taskiq + Redis)
- Responsibilities: Execute long-running jobs (e.g., refresh Facebook tokens)

## Error Handling

**Strategy:** Exception hierarchy with domain exceptions propagated to HTTP responses

**Patterns:**

- **Domain Exceptions:** Base class `AuthDomainException` and subclasses (e.g., `InvalidCredentialsException`, `EmailNotVerifiedException`, `OAuthStateInvalidError`)
  - Raised in use cases
  - Caught by exception handlers in middleware
  - Mapped to appropriate HTTP status codes

- **Infrastructure Exceptions:** SQLAlchemy `IntegrityError` for duplicate keys, constraint violations
  - Caught and converted to domain-level HTTP responses
  - Example: Duplicate email → `409 Conflict`

- **Generic Exception Handler:** Catches unexpected exceptions in `apps/api/src/prosell/infrastructure/api/middleware/exception_handlers.py`
  - Logs error with request context
  - Returns `500 Internal Server Error` in production

**Exception Handler Registration:**
- Location: `apps/api/src/prosell/infrastructure/api/main.py` (lines 44-46)
- Pattern: FastAPI `add_exception_handler()` for each exception type

## Cross-Cutting Concerns

**Authentication:**
- JWT-based with HTTP-only cookies
- Middleware in `apps/api/src/prosell/infrastructure/api/middleware/auth_middleware.py`
- Current user injected into FastAPI dependencies
- Supports optional authentication (public endpoints)

**Authorization (RBAC):**
- Role-based access control defined in domain
- Roles: SUPER_ADMIN, ADMIN, MANAGER, SALES_AGENT, SALES_USER, VIEWER
- Permissions mapping in `apps/api/src/prosell/domain/entities/role.py` (ROLE_PERMISSIONS)
- Checked by decorators/middleware (not yet fully implemented)

**Rate Limiting:**
- Library: SlowAPI (token bucket algorithm)
- Configuration: `apps/api/src/prosell/infrastructure/api/middleware/rate_limit_middleware.py`
- Applied per endpoint (e.g., `@limiter.limit("5/minute")`)

**Security Headers:**
- Middleware in `apps/api/src/prosell/infrastructure/api/main.py` (lines 77-119)
- Includes: X-Content-Type-Options, X-Frame-Options, CSP, STS, Referrer-Policy, Permissions-Policy
- Known issue: CSP uses `unsafe-inline` (tracked in issue SECURITY-001)

**Logging:**
- Standard Python logging module
- Configured in `apps/api/src/prosell/core/config.py`
- Captured in middleware and exception handlers
- SQL queries logged when `debug=True` and `environment=development`

**Validation:**
- Input validation via Pydantic in DTOs
- Business rule validation in domain entities and use cases
- Database constraints (unique, foreign keys, NOT NULL)

**Multi-Tenancy:**
- All aggregates include `tenant_id` field
- `Organization.id == Organization.tenant_id` (self-reference)
- Isolation enforced via queries filtering by tenant_id
- No cross-tenant data access (business rule in repositories)

## Database Architecture

**ORM:** SQLAlchemy 2.0 with async support (asyncpg driver)

**Models Location:** `apps/api/src/prosell/infrastructure/models/`

**Key Features:**
- Base declarative model: `Base` from `apps/api/src/prosell/infrastructure/database/base.py`
- Mapped columns using `Mapped[]` type hints and `mapped_column()`
- Async queries using `select()` and `await session.execute()`
- Relationships lazy-loaded (not eager-loaded)
- Soft deletes via `archived_at` timestamp (not via deleted flag)

**Migrations:** Alembic in `apps/api/alembic/`
- Versioned migrations tracked in `versions/` directory
- Migration workflow: Create entity → Create migration → Run migration → Implement repository
